# 芯片 Datasheet QA Agent 需求 Spec

## 1. 产品目标

构建一个本地芯片资料问答系统，支持用户基于本地 PDF / PPT datasheet、联网搜索结果以及芯片基本常识，对指定芯片型号进行技术问答。

系统回答问题时必须遵循：

1. 先确认或识别目标芯片型号。
2. 优先基于本地资料回答。
3. 可补充联网搜索和芯片领域常识。
4. 回答先给结论。
5. 再给支持论据，包括：
   - 本地资料中的原文引用
   - 联网资料中的来源引用
   - 基于芯片常识的推导逻辑
6. 明确区分：哪些是资料原文，哪些是推导，哪些是外部信息。
7. 如果资料不足，必须说明不确定性，不能硬答。

## 2. 目标用户

主要用户是需要快速理解芯片资料的人，例如：

- 芯片产品经理
- 硬件工程师
- 嵌入式工程师
- FAE / AE
- 技术销售
- 方案工程师
- 采购或选型人员

## 3. 典型使用方式

用户提问时优先指定芯片型号，再描述问题。

示例：

```text
芯片型号：XXX1234
问题：这个芯片是否支持 I2C？最高速率是多少？
```

或：

```text
请基于 ABC-9000 这个芯片回答：它的工作电压范围是多少？是否适合 3.3V 系统？
```

系统需要识别：

- 目标芯片型号
- 问题意图
- 需要检索的资料范围
- 是否需要联网搜索
- 是否需要进行推导

## 4. 输入范围

### 4.1 本地文件

支持：

- PDF datasheet
- PDF application note
- PDF user manual
- PPT / PPTX 产品介绍
- PPT / PPTX 培训材料

后续可扩展：

- Excel 选型表
- Word 文档
- HTML 数据手册
- Markdown 笔记

### 4.2 用户输入

用户输入应包含：

- 芯片型号，优先级最高
- 技术问题
- 可选约束条件，例如应用场景、供电电压、接口需求、封装要求、温度范围等

## 5. 输出格式

回答必须采用固定结构。

```markdown
## 结论

直接回答问题。必要时说明确定性等级。

## 依据

### 1. 本地资料证据

- 来源文件：
- 页码 / Slide：
- 原文摘录：
- 对应解释：

### 2. 联网资料证据

- 来源链接：
- 摘要：
- 与问题的关系：

### 3. 芯片常识与推导

- 已知条件：
- 推导过程：
- 推导结论：

## 不确定性与限制

说明哪些信息资料中没有明确写出，哪些结论是推导得到的。

## 建议

如果适用，给出工程建议、选型建议或下一步验证建议。
```

## 6. 信息优先级

回答时的信息优先级如下：

1. 用户明确指定的本地 datasheet / PPT
2. 同型号芯片的其他本地资料
3. 芯片厂商官网资料
4. 可信第三方资料
5. 通用芯片设计 / 硬件常识
6. 模型自身知识

如果不同来源冲突，优先级高的来源胜出，同时说明冲突。

示例：

> 本地 datasheet 标称工作电压为 2.7V-5.5V，而某第三方页面写为 3.0V-5.5V。由于 datasheet 是一手资料，因此以本地 datasheet 为准。

## 7. 核心功能需求

### 7.1 文件导入

系统需要支持导入本地 PDF / PPT 文件，并提取：

- 文本内容
- 页码 / slide 编号
- 表格内容
- 标题层级
- 图片附近说明文字
- 芯片型号
- 文档类型
- 厂商名称
- 版本号 / 日期

重点：必须保留来源定位信息。

示例结构：

```json
{
  "file_name": "ABC1234_Datasheet.pdf",
  "chip_model": "ABC1234",
  "page": 12,
  "section": "Electrical Characteristics",
  "text": "Operating supply voltage: 2.7V to 5.5V"
}
```

### 7.2 芯片型号识别

系统要从文件名、正文、用户问题中识别芯片型号。

需要处理：

- 一个文件对应一个芯片
- 一个文件包含多个芯片型号
- 用户输入型号和文件型号模糊匹配
- 型号存在别名、后缀、封装变体

示例：

```text
ABC1234
ABC1234A
ABC1234-QFN
ABC1234-TR
```

系统需要判断这些是否属于同一芯片系列，必要时提醒用户确认。

### 7.3 检索能力

检索时不能只做全库搜索，必须优先按芯片型号过滤。

推荐流程：

```text
用户问题
-> 提取芯片型号
-> 匹配本地资料
-> 在对应芯片资料内检索
-> 如果结果不足，再扩展到同系列资料
-> 如果仍不足，触发联网搜索
```

检索内容包括：

- 参数表
- Pin description
- Electrical characteristics
- Absolute maximum ratings
- Functional description
- Register map
- Timing diagram
- Application circuit
- Ordering information
- Package information

### 7.4 回答生成

回答生成必须遵守：

- 先结论
- 后证据
- 必须引用原文
- 推导逻辑单独说明
- 不确定就说明不确定
- 不允许把推导包装成原文结论
- 不允许没有证据地给确定性结论

### 7.5 联网搜索

联网搜索不是默认滥用，而是在这些场景触发：

- 本地资料没有相关内容
- 本地资料疑似过旧
- 用户明确要求联网查证
- 需要查询厂商官网最新资料
- 需要确认停产、替代料、价格、供应状态等时效信息
- 需要查 errata、application note、reference design

联网结果需要标记来源：

```text
来源类型：厂商官网 / 分销商 / 第三方博客 / 论坛 / 其他
可信度：高 / 中 / 低
```

### 7.6 芯片常识推导

允许使用芯片常识，但必须显式标注。

适合推导的场景：

- 根据电压范围判断是否适合 3.3V 系统
- 根据接口类型判断是否需要上拉电阻
- 根据封装和功耗估计散热风险
- 根据 ADC 分辨率和参考电压推导 LSB
- 根据时钟频率推导理论吞吐
- 根据 absolute maximum ratings 提醒不能作为工作条件

禁止做法：

- 把常识推导说成 datasheet 明确说明
- 在没有证据时给强确定性结论
- 忽略 datasheet 中的限制条件

## 8. 非功能需求

### 8.1 准确性

必须可溯源，优先引用原文。

### 8.2 可解释性

回答要区分：

- 原文证据
- 联网证据
- 模型推导
- 不确定信息

### 8.3 安全性

不能擅自给出高风险工程建议，例如超规格使用芯片。

### 8.4 可维护性

后续应支持新增芯片资料、重新索引、版本管理。

### 8.5 可评测性

需要设计测试集，例如：

- 参数问答
- 兼容性判断
- 接口支持判断
- 功耗计算
- 封装/引脚查询
- 寄存器说明查询
- 资料缺失时的拒答

## 9. 行动计划

### 阶段 1：需求澄清与样本准备

先收集 5-10 份代表性资料：

- PDF datasheet
- PPT 产品介绍
- 长文档
- 含表格文档
- 含多芯片型号文档
- 中英文混合文档

同时定义 20-30 个典型问题，例如：

- 这个芯片的工作电压范围是多少？
- 是否支持 SPI / I2C / UART？
- 最大工作温度是多少？
- 某个 pin 的功能是什么？
- 这个芯片能否用于 3.3V 系统？
- ADC 分辨率是多少？
- 是否有低功耗模式？
- 某个寄存器 bit 的含义是什么？

### 阶段 2：设计资料解析方案

重点不是马上做 RAG，而是先把文档结构化。

需要确定：

- PDF 文本如何提取
- PPT slide 如何提取
- 表格如何提取
- 页码和 slide 编号如何保留
- 图片 / 图表暂时如何处理
- 每个 chunk 如何绑定来源

初版可以先处理文本和表格，图片里的信息后续再扩展 OCR 或多模态。

### 阶段 3：设计索引结构

建议建立两层索引。

文档元数据索引：

- chip_model
- vendor
- document_type
- file_name
- version
- page_count
- language

内容向量索引：

- chunk_text
- chip_model
- file_name
- page_or_slide
- section_title
- document_type
- embedding
- keywords

这样可以做到：

```text
先按芯片型号过滤，再做语义检索。
```

### 阶段 4：设计问答流程

推荐问答流程：

```text
用户问题
-> 解析芯片型号和问题意图
-> 检查是否匹配到本地资料
-> 在目标芯片资料中检索
-> 判断检索结果是否足够
-> 如果不足，决定是否联网搜索
-> 汇总本地证据、联网证据、常识推导
-> 生成“先结论后论据”的回答
-> 输出不确定性和建议
```

### 阶段 5：定义回答规范

必须包含：

- 结论
- 本地资料证据
- 原文摘录
- 推导逻辑
- 不确定性
- 建议

拒答规则：

- 没有找到资料时不能编
- 找到的资料和问题无关时不能硬答
- 资料冲突时必须说明冲突
- 只有常识推导时必须降低确定性

### 阶段 6：设计评测集

建议至少准备 50 个问题：

- 30 个资料中能直接回答的问题
- 10 个需要简单推导的问题
- 5 个资料中没有答案的问题
- 5 个容易混淆芯片型号的问题

评测指标：

- 是否识别正确芯片型号
- 是否检索到正确页码 / slide
- 是否引用了原文
- 结论是否正确
- 推导是否合理
- 是否正确表达不确定性
- 是否错误使用联网信息覆盖本地资料

### 阶段 7：代码实现顺序

等 Spec 确认后，代码可以按这个顺序实现：

1. 文档解析
2. 芯片型号识别
3. 元数据索引
4. 向量检索
5. 本地资料问答
6. 回答格式约束
7. 联网搜索补充
8. 常识推导模块
9. 评测脚本
10. 简单界面

## 10. MVP 范围

第一版建议只做到：

- 支持 PDF / PPT 文本解析
- 支持按芯片型号过滤
- 支持本地资料 RAG
- 回答中包含结论、原文引用、推导逻辑
- 暂时不自动联网，只在本地资料不足时提示“建议联网补充”

第二版再加：

- 联网搜索
- 表格增强
- OCR / 图片理解
- 多芯片型号消歧
- 评测体系
- 前端界面

# Agent 开发学习内容展开格式

后续每个学习主题都按 `What -> Why -> How` 三个维度展开。

这个格式的目的，是避免只停留在“知道一个概念”或“照着代码跑通 Demo”，而是把每个知识点都学成可解释、可判断、可实现、可面试表达的能力。

## 总体结构

每个主题统一拆成三部分：

1. What：它是什么，核心原理是什么
2. Why：为什么需要它，它解决什么问题，适用于什么场景
3. How：工程上如何实现，代码怎么写，如何验证效果

推荐每个主题最终都产出四样东西：

- 一段自己的概念解释
- 一张流程图或架构图
- 一段可运行代码
- 一段面试表达

## 1. What：概念与原理

这一部分回答：这个东西到底是什么？

需要覆盖：

- 核心定义
- 关键组成部分
- 工作流程
- 输入和输出
- 与相近概念的区别
- 常见误区

建议问题清单：

- 它解决的是哪一类问题？
- 它的最小工作单元是什么？
- 它依赖哪些前置能力？
- 它的输入是什么，输出是什么？
- 它内部通常经历哪些步骤？
- 它和相似概念有什么区别？

示例：学习 RAG 时，What 部分需要讲清楚：

- RAG 是 Retrieval-Augmented Generation
- 它由文档处理、向量化、检索、上下文构造、生成回答组成
- 输入是用户问题和文档知识库
- 输出是基于检索上下文生成的回答
- 它和微调不同，RAG 主要解决外部知识注入和知识更新问题

## 2. Why：目的、价值与适用场景

这一部分回答：为什么要用它？不用会怎样？什么时候该用，什么时候不该用？

需要覆盖：

- 业务价值
- 技术价值
- 适用场景
- 不适用场景
- 成本和收益
- 风险与限制
- 在大厂真实业务中的意义

建议问题清单：

- 它相比直接调用 LLM 多解决了什么？
- 它改善了准确率、效率、成本、体验还是可控性？
- 它适合哪些业务场景？
- 它不适合哪些场景？
- 它的主要风险是什么？
- 上线后怎么衡量它是否有效？

示例：学习 Tool Calling 时，Why 部分需要讲清楚：

- LLM 本身不能直接访问实时数据或执行业务动作
- Tool Calling 让模型可以调用搜索、数据库、CRM、邮件、工单等外部能力
- 它适合查询、创建、修改、审批等流程型任务
- 高风险操作必须加入权限、确认和审计
- 衡量指标包括工具调用成功率、参数正确率、任务完成率、人工接管率

## 3. How：工程实践与代码实现

这一部分回答：工程里怎么落地？代码怎么组织？如何验证它真的有效？

需要覆盖：

- 最小可运行实现
- 推荐项目结构
- 核心代码路径
- API 设计
- 数据结构设计
- 异常处理
- 日志与监控
- 测试与评测
- 常见坑和排查方法

建议问题清单：

- 最小 Demo 怎么写？
- 生产版本需要哪些模块？
- 关键数据结构是什么？
- 如何处理失败、超时、空结果、模型输出格式错误？
- 如何写测试？
- 如何记录日志和 trace？
- 如何评估效果？
- 如何把它接入主项目？

示例：学习 LangGraph Agent 时，How 部分需要落到：

- 定义 state schema
- 定义 node 函数
- 定义条件边
- 接入 LLM 和 tools
- 保存 checkpoint
- 编写一个 `/agent/run` API
- 写一个失败重试和人工确认节点
- 准备几个测试用例验证流程是否正确

## 学习记录模板

每次学完一个主题后，按下面格式记录。

```markdown
# 学习记录：主题名称

日期：YYYY-MM-DD
投入时间：X 小时

## 我理解的 What

用自己的话解释，不照抄资料。

## 我理解的 Why

说明它解决的问题、适合场景和不适合场景。

## 我实现的 How

记录代码位置、接口、运行方式、关键实现。

## 遇到的问题

记录 bug、困惑、设计取舍。

## 验证结果

记录测试结果、评测指标、截图或日志。

## 面试表达

用 3-5 句话把这个主题讲给面试官。

## 下一步

列出需要继续补的内容。
```

## 使用原则

1. 先能解释，再写代码。
2. 代码必须能运行，不能只停留在伪代码。
3. 每个主题都要沉淀到主项目里。
4. 每个主题都要准备一段面试表达。
5. 学习材料可以很多，但最终沉淀必须简洁。
6. 遇到复杂概念时，优先画流程图，再写代码。
7. 每周复盘时检查：本周的 What、Why、How 是否都讲得清楚。
